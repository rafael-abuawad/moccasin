# Documentation style guide

Use this guide when editing user-facing prose in the moccasin repository (Sphinx `.rst`, root markdown, and ancillary READMEs).

## Product and tooling names

| Context | Use |
|---------|-----|
| The framework (titles, marketing) | **Moccasin** |
| Package, config files, narrative | **moccasin** |
| CLI command | **mox** |
| CLI help and reference sections | **Moccasin CLI** |

## Ecosystem terms

| Topic | Convention |
|-------|------------|
| ZK rollup stack | **ZKsync** in prose; `zksync` only in code, TOML keys, and URLs |
| Apple desktop OS | **macOS** (not MacOS) |
| Generic Unix-like OS | **Linux** |
| Onchain (adjective/adverb in prose) | **onchain** (one word, not on-chain) |

Proper nouns to spell correctly: **Titanoboa**, **Mamushi**, **Hypothesis**, **Anvil**, **Vyper**, **Jupyter**.

## Voice and structure

- Prefer second person (**you**) in tutorials and how-tos.
- Follow [Diátaxis](https://diataxis.fr/): do not turn tutorials into reference pages or vice versa.
- Keep one canonical explanation for large tables (e.g. TOML parameter matrices); link from overview pages instead of duplicating.

## Generated content

Do not hand-edit prose that is generated from code or build scripts unless the generator is wrong:

- CLI help blocks via `docs/source/moccasin_wrapper_for_docs.py` and Jinja in `index.rst` — fix strings in Python, then rebuild.
- `docs/source/_autogenerated/*.rst` — fix `docs/source/_generate_vars.py` if needed.

## Local checks (no CI yet)

```bash
uv sync --extra docs
just docs
```

Optionally run [typos](https://github.com/crate-ci/typos) locally before opening a PR. Repository automation (typos CI, `sphinx-build -b linkcheck`) is tracked separately; see the documentation quality tracking issue.
